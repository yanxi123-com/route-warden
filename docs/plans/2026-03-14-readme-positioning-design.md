# Route Warden README 定位设计

日期：2026-03-14  
项目：`route-warden`

## 1. 目标

将 `README.md` 首页从“开发中 MVP 功能清单”改为“先打动用户、再给出使用说明”的结构，突出 Route Warden 的核心价值：

- 不是全局固定一个节点
- 而是针对不同站点/业务组，自动切换到各自更稳定的节点

## 2. 需要强调的核心信息

首页应让用户在最短时间内理解以下几点：

1. `route-warden` 是给 Mihomo/Clash 用的自动切换守护程序。
2. 它按站点或业务组做独立探测与切换，而不是“全局选一个最快节点”。
3. 典型场景需要明确写出：
   - `OpenAI` 可能在 A 节点更稳
   - `GitHub` 可能在 B 节点更稳
   - 其他网站继续走各自更合适的线路
4. 它会持续探测、评分、防抖切换，并把状态写入 SQLite，适合常驻运行。

## 3. 建议结构

### 3.1 顶部定位

用一句产品化表述开场，例如：

“让不同站点自动走各自更稳定的节点，而不是所有流量都绑在同一个节点上。”

紧接一段 2-4 句的说明，明确：
- `OpenAI`、`GitHub`、普通网站可能各自适合不同节点
- Route Warden 会持续探测并自动切换对应策略组

### 3.2 场景说明

增加一个“典型场景”或“它解决什么问题”小节，说明：
- 手动切节点很麻烦
- 一个节点不可能对所有站点都最优
- 频繁手切容易打断工作

### 3.3 能力摘要

将现有 MVP 清单改写为“用户收益导向”的能力摘要，例如：
- 按组探测目标站点可用性
- 自动切换到更稳节点，并避免抖动
- 持久化记录每轮探测和切换事件
- 支持常驻模式与 Clash Verge 配置同步

### 3.4 使用说明

保留原 README 的实用内容，但顺序后移：
- 快速开始
- `sync-rw-profile` 用途说明
- 默认配置路径
- macOS `launchd` 常驻步骤
- 延伸文档链接

## 4. 文案边界

需要避免的误导：

- 不要写成“智能加速所有网站”
- 不要暗示“绝对最快”
- 不要把“分站点独立切换”说成“自动全局选最佳线路”

推荐表述：

- “更稳定”
- “更合适”
- “按不同站点独立决策”
- “减少手动切节点”

## 5. 验收标准

1. 用户打开 README 前 10 秒内能理解“不同站点自动切不同节点”。
2. README 中明确出现 `OpenAI` / `GitHub` / 其他站点的示例。
3. 原有快速开始和常驻运行说明仍保留。
4. README 语气更像产品介绍，同时不牺牲操作可用性。
